## 核心结论（先读这段）

- **做什么**：把 BOOT / APP / Pflash 三个固件文件合并为一个 `Full.hex`，核心是**仲裁相同地址的冲突**（如 BOOT 与 APP 的中断向量表、配置区重叠）。
- **可行性**：高。核心技术（Intel HEX 解析、字节级冲突检测、合并）都是成熟确定性算法。三个输入文件已**实测验证**。
- **关键决策**：① **字节级地址模型** + ② **WinMerge 风格界面** + ③ **.NET Framework 4.8 WPF** + ④ **CommunityToolkit.Mvvm**（微软官方 MVVM 库）。
- **目标机**：Win10/11（4.8 预装，免部署）。
- **工作量**：约 4–6 人天。

## 一、已验证的关键事实（实测三个输入文件）

| 文件 | 格式 | 地址范围 | 段数 | 校验和 |
|---|---|---|---|---|
| `7504_FOC-APP.hex`（APP） | I16HEX（类型02×5、**03×1**） | 0x0 ~ 0x80FFF | 3 段（向量表/主程序/配置区） | 全通过 |
| `7504ProjectRam-BOOT.hex`（BOOT） | I16HEX（类型02×2、**03×1**） | 0x0 ~ 0x80FFF | 2 段（启动代码/配置区） | 全通过 |
| `DFLASH_*.dat`（Pflash） | **二进制 raw（非 HEX）**，基地址 0xE00000 | 0xE00000 ~ 0xE0AFFF | 45KB，**几乎全 0xFF**（仅 5 字节有效） | — |

**实测冲突（仅 2 处）**：
1. BOOT[0–0xBFFF] ∩ APP[0–0x123] = **部分重叠 292B**（APP 向量表落在 BOOT 代码区头部）
2. BOOT[0x7F000–0x80FFF] ∩ APP[0x7F000–0x80FFF] = **完全重叠 8KB**（配置/校验区）
3. Pflash 地址空间（0xE00000）与 BOOT/APP **完全独立，无冲突**。

> 需求说明书假设三个都是 HEX，但 Pflash 实际是 `.dat` 二进制；需求遗漏了**记录类型 03**。两点已在设计中扩展。

## 五、核心数据模型（字节级）

采用**字节级地址模型**（非需求原文的"段级"），修复需求 4.2/4.4 对"部分重叠"失效的漏洞：

- `MemoryImage`：统一字节地址空间，每个地址记录「字节值 + 来源文件」
- 冲突地址 = 同一地址有 ≥2 个来源且值不同
- 段归并（需求 3.1.3）作内部步骤，用于冲突判定，不作 UI 操作单位
- **仲裁方式**：参考 WinMerge「逐块人工选择」——冲突地址高亮，用户逐块决定保留谁；「文件优先级」仅作一键默认填充，可微调

## 七、功能详述

**解析**：I8/I16/I32，类型 00/01/02/04/05 + **03 容错忽略**，未列举类型容错；逐行校验和（失败报行号）；02（左移4位）/04（左移16位）地址换算；**05 起始执行地址：记录到 `MemoryImage.StartLinearAddress`（不作为数据合并），合并输出时在 EOF 前保留**；DAT 手动输入基地址（文件名 `0xE00000` 预填可改）。**流式读取**（`HexParser` 用 `StreamReader` 逐行、`BinaryDatParser` 用 `FileStream` 分块，避免大文件一次性载入 OOM）。

**比对**：多文件映射统一地址空间，逐地址判定 相同 / 冲突 / 独占；冲突即需求的"完全/部分/包含"重叠（字节级统一处理）。

**合并**：优先级下拉 + 冲突行逐块改选，字节级覆盖；输出 I32HEX（按地址排序、16字节/行、类型04 + 起始地址05、末尾 `:00000001FF`）；空隙默认填 0xFF，**填充阈值 1MB**（单段间隙超过 1MB 视为跨大区不填，避免 HEX 输出膨胀 + 32 位进程 OOM）；**到 DFlash 的间隙自动识别**（DAT 基地址 = DFlash 起点，落点跨 DFlash 的间隙默认忽略，并在预览/状态栏提示）；**逐字节回读校验**（输出再解析与选中数据比对，零差异才放行）。**解析后台化**（`Task.Run` 后台解析 + `Dispatcher` 回 UI 线程构建，防大文件 UI 假死）。

## 十三、变更日志

### 2026-07-01 ~ 07-02（连续迭代）

**功能增强**
- **05 起始执行地址**：`HexParser` 解析时记录到 `MemoryImage.StartLinearAddress`（不作数据合并）；`HexWriter.Write` 加重载，EOF 前输出类型 05 记录；`CompareViewModel.PickStartLinearAddress` 按优先级取首个含 05 文件入口。
- **到 DFlash 间隙识别**：`Load` 扫描 `.dat` 取基地址最小值作 DFlash 起点；`ComputePreview`/`ApplyGapFill` 对落点跨 DFlash 的间隙默认忽略并单独提示（预览浅蓝块 + 状态栏）。
- **间隙填充阈值**：定 1MB（`GapFillMaxBytes`）。曾抬到 32MB 导致 Full.hex 膨胀到 41MB + 32 位进程 OOM，回退 1MB，并在注释写明两道硬约束。
- **菜单重组**：顶级 5→4（文件/合并/导航/帮助），优先级·空隙填充收进「合并」；新增「帮助 → 关于」（HexMerge V版本）。
- **合并预览**：段与间隙各自成行、统一对齐，间隙行末标「(间隙)」。

**健壮性 / 性能**
- **流式解析**：`HexParser` 改 `StreamReader`、`BinaryDatParser` 改 `FileStream` 64KB 分块，去掉 `ReadAllLines/ReadAllBytes` 全量载入，防大文件 OOM。
- **解析后台化**：`CompareViewModel.Load` 改 `Task.Run` 后台解析 + `Dispatcher` 回 UI 线程构建；加 `Loaded` 事件，`CompareWindow` 订阅后 `BuildMenus` + 刷新布局，防大文件 UI 假死。
- **FindUnit 字典化**：新增 `_unitByAddr`，`FindUnit`/`CycleChoice` 从 O(n) 线性改 O(1)。
- **定位栏**：合并连续同色段绘制，`DrawRectangle` 从行数级降到颜色变化级。
- **渲染缓存**：表头段信息 `FormattedText` 预建（`_segText`），消除每帧分配。

**体验 / 代码质量**
- 比较视图键盘 ↑/↓ 导航 + 视口跟随。
- 提取 `Core/FirmwareLoader.cs`（HEX/DAT 分发 + `IsDat`），消除两 VM 重复。
