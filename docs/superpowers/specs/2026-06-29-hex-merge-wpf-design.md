# HEX 文件合并工具 — 设计规范 (Spec)

> 日期：2026-06-29

> 目标平台：Windows 桌面（WPF, .NET Framework 3.5）

---

## 一、项目目标

将三个独立编译的固件 HEX 文件（BOOT / APP / Pflash）合并为一个完整的 `Full.hex`。

核心难点：不同文件在相同地址区间存在冲突（典型如 BOOT 与 APP 的中断向量表、配置区重叠），需人工仲裁保留哪个文件的数据。本工具解决「多个独立编译的固件模块合并烧录时，手动仲裁地址冲突」的问题。

## 二、范围（MVP）

### 2.1 MVP 包含
- **解析文件**：加载并解析 HEX（I8/I16/I32）与二进制 DAT 文件
- **比对文件**：WinMerge 风格多文件（2–3 个）地址对比，差异高亮、缺失标灰
- **合并文件**：按优先级或逐块改选，输出 `Full.hex`（I32HEX）
- **逐字节回读校验**（稳定性保障）

### 2.2 非目标（后续迭代）
- 图形化地址轴视图（需求 3.2.2）
- 段列表表格视图（需求 3.2.1，以字节比较视图替代）
- 一键策略矩阵（仅 BOOT / 仅 APP / 仅 Pflash / 自动去重，降为优先级下拉）
- 导出前预览弹窗（降为状态栏摘要）
- 段级勾选 / 全选 / 反选操作

## 五、架构

采用**单工程**结构（不拆 Core / App / Fw / Dll / Lib 独立工程）：全部代码（Core 算法 + 手写 MVVM + UI）集中在一个 WPF 应用工程 `ExeProject/HexMerge` 内，按文件夹分层；测试单独一个工程。

```
D:\Software\FTHexMerge\
├─ FTHexMerge.sln
├─ Bin.Net\                      ExeProject 编译输出（exe + dll）
├─ Lib.Net\nunit.framework.dll   NUnit 2.6.x（测试依赖）
├─ Doc\                          需求 / 可行性 / spec
└─ Src\
   ├─ ExeProject\HexMerge\       (WPF 应用，单工程，含全部代码)
   │   ├─ Core\        算法：HexParser, BinaryDatParser, ConflictDetector,
   │   │                                    Arbiter, HexWriter, RoundTripVerifier
   │   ├─ Models\      MemoryImage, Segment, ByteSource
   │   ├─ Mvvm\        手写基类：ObservableObject, RelayCommand
   │   ├─ ViewModels\  FileSelectViewModel, CompareViewModel
   │   ├─ Views\       FileSelectWindow, CompareWindow
   │   ├─ Controls\    AddressCompareGrid, LocationPanel
   │   └─ App.xaml / MainWindow.xaml
   └─ TestProject\HexMerge.Tests\ (NUnit 2.6.x，引用 HexMerge)
```

**分层原则**：`Core\` 文件夹为纯 C# 算法（不引用 WPF），可被 TestProject 直接测试；`Views\` / `ViewModels\` 负责交互。
**MVVM**：手写 `ObservableObject`（实现 `INotifyPropertyChanged`）+ `RelayCommand`（实现 `ICommand`），不引入第三方 MVVM 库。
**输出**：ExeProject 输出指向 `Bin.Net\`。

## 六、核心数据模型（字节级）

采用**字节级地址模型**（而非需求原文的"段级"），修复需求 4.2 / 4.4 对"部分重叠"失效的逻辑漏洞。

- `MemoryImage`：统一字节地址空间，每个地址记录「字节值 + 来源文件」
- 冲突地址 = 同一地址有 ≥2 个来源且值不同
- 段归并（需求 3.1.3）作为内部步骤，用于性能优化与冲突判定，不作为 UI 操作单位
- 内部存储：按段 / 区间组织 + 冲冲位图（196KB 规模无压力）

## 七、界面设计

### 7.1 第一级：文件选择窗（启动时）
- **多行纵向排列**（不是多列）：文件1、文件2、文件3 各占一行
- 每行：路径输入框 + [浏览…] 按钮，**支持拖拽文件到输入框**
- 文件3 为可选（至少填 2 个）
- [比较] 进入比较视图；[取消] 退出

```
┌─ HEX 文件合并工具 ──────────────────────┐
│  第一个文件（左）：                  │
│  [                       ] [ 浏览… ]   │
│  第二个文件：                       │
│  [                       ] [ 浏览… ]   │
│  第三个文件（可选）：               │
│  [                       ] [ 浏览… ]   │
│              [ 比较 ]   [ 取消 ]      │
└────────────────────────────────────────┘
```

### 7.2 第二级：比较视图
- 从左到右：**定位栏 | 文件1 | 文件2 |（文件3）**
- 列数随填入文件数动态变化（2 或 3）
- 每行：**地址 + 16 字节值**（hex dump 风格）
- **差异高亮**：同地址不同值 → 红
- **缺失标灰**：某文件无此地址 → 该列标灰占位
- **同步滚动**：按地址对齐，缺失处插占位行
- **控件**：每列用 `ListView`+`GridView`（3.5 无 `DataGrid`），`VirtualizingStackPanel` 虚拟化保证约 1.2 万行 / 文件流畅
- **定位栏**：鸟瞰差异分布，点击跳转

### 7.3 菜单与工具栏（参考 WinMerge 精简）
- 文件：重新比较、保存合并、退出
- 视图：上一差异、下一差异
- 工具栏：优先级下拉（默认 BOOT > APP > Pflash）、合并按钮
- 帮助：关于

## 八、功能详述

### 8.1 解析
- 支持 I8 / I16 / I32 HEX，记录类型 00 / 01 / 02 / 04 / 05
- **类型 03（Start Segment Address）容错忽略**（记录但不参与合并）；未列举类型容错不报错
- 逐行校验和，失败报错并标行号；格式异常（长度不符 / 非法字符）报行号
- 类型 02（左移 4 位）/ 04（左移 16 位）地址换算，互斥正确
- 二进制 DAT：用户手动输入基地址（文件名 `0xE00000` 预填可改）

### 8.2 比对
- 多文件映射到统一地址空间，逐地址判定：相同 / 冲突（多源不同值）/ 独占（仅一个源）
- 冲突即需求的"完全 / 部分 / 包含"重叠（字节级统一处理）

### 8.3 合并
- 优先级下拉（默认 BOOT > APP > Pflash，可切换）+ 冲突行可逐块改选
- 逐字节决定保留谁（字节级覆盖）
- 输出 I32HEX：按地址排序，16 字节 / 行，类型 04 扩展线性地址，末尾追加 `:00000001FF`（EOF）
- 空隙填充：默认 0xFF（不填 / 0x00 选项后续加）
- **逐字节回读校验**：输出文件再解析，与选中数据逐字节比对，零差异才放行

## 十、测试策略（TDD）

- `Core` 100% 覆盖，**先写测试再实现**
