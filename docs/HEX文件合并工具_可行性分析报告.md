## 二、输入文件实测情况

已实际解析全部三个文件，真实数据如下：

| 文件 | 实际格式 | 真实地址范围 | 大小 / 段数 | 校验和 |
|---|---|---|---|---|
| `7504_FOC-APP.hex`（APP） | **I16HEX**（含类型02×5） | `0x00000000 ~ 0x00080FFF` | 196 KB / **3 段** | ✅ 全通过 |
| `7504ProjectRam-BOOT.hex`（BOOT） | **I16HEX**（含类型02×2） | `0x00000000 ~ 0x00080FFF` | 57 KB / **2 段** | ✅ 全通过 |
| `DFLASH_0xE00000_0xE0AFFF.dat`（Pflash） | **⚠️ 二进制 raw，非 HEX** | `0x00E00000 ~ 0x00E0AFFF` | 45 KB / 1 段 | — |

### 真实地址段分布

```
BOOT:  ████████████████ 0x00000000–0x0000BFFF (49KB 启动代码+向量表)
                         ░░░░░░░░░░░░░░░░░░░░░ (空隙)
                         ░░░░ 0x0007F000–0x00080FFF (8KB 配置/校验区)

APP:   ░ 0x00000000–0x00000123 (292B 中断向量表)
            ░░░░░░░░░░░░░░░░ 0x0000C000–0x00039FFF (184KB 主程序)
                         ░░░░ 0x0007F000–0x00080FFF (8KB 与BOOT同址!)

Pflash(0xE00000):         独立地址空间，与上两者完全不重叠，内容全 0xFF
```

### 实测真实冲突（仅 2 处，边界清晰、规模小）

1. `BOOT[0x0–0xBFFF]` 与 `APP[0x0–0x123]` → **中断向量表地址冲突**（APP 向量表落在 BOOT 代码区开头，部分重叠）
2. `BOOT[0x7F000–0x80FFF]` 与 `APP[0x7F000–0x80FFF]` → **配置/校验区完全重叠**

## 五、核心技术难点与风险

1. **I16HEX → I32HEX 的地址换算正确性**（重点测试项）
   类型02「左移4位」与类型04「左移16位」混用时要保证换算零误差。建议以**逐字节回读校验**作为验收标准（输出再解析，与输入逐字节比对）。

2. **图形视图的可读性**
   地址跨度达 `0x00000000 ~ 0x00E0B000`（约 14 MB），而段大小从 292 字节到 184 KB 不等。线性轴会让小段不可见。需要支持缩放/分段绘制——这是体验成败点。

3. **Pflash DAT 全是 0xFF**
   当前示例的 Pflash 内容是擦除态空数据，合并它意义有限。**需向固件方确认**：是占位示例，还是真实场景下会有有效数据？（不影响工具开发，但影响验收数据）

4. **向量表冲突的仲裁语义**
   实测冲突①是「部分重叠」（APP 向量表在 BOOT 代码区头部）。仲裁时是按**整个段**二选一，还是允许**按地址区间切分**保留各自有效部分？需求倾向整段二选一（更简单），需确认这符合实际烧录逻辑。
